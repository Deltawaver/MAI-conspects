### Начало
C# компилируется в промежуточный код (IL). Можем запускать один и тот же скомпилированный код на любой машине.

Как это работае?
C# -> [коммпилятор] -> IL (.dll, .exe (небинарный код - там инструкция по запуску ВМ)) -> [CLR, JIT] ->Машинный код

ВМ привносит сборщик мусора. 
CLR - управляет памятью, отвечает за бесопасность типов, обработку ошибок, загрузку типов, классов и библиотек и упровления потоками.
JIT - отвечает за перевод CLR в машинный язык.

### Garbage Collector
см доклад от Станислава Сидристый.
GC Roots - список "точно живых" объектов. Все остальное - мусор

Всего 3 кучи:
1. Small Object Heap
2. Large Object Heap (>85 000 Кб)
3. Pinned Object Heap
(якобы вопрос на собесе)

Для удаления запустили механизм поколений.
Сборка мусора запускается по поколениям (0, 1, 2). Данные пережили зачистку поколений -> поднимается на поколение выше. Если очистка всех поколений не помогла - "out of memroy exception". Чистим поколение до тех пор, пока нужное кол-во памяти не освободится т.е. мусор ещё может остаться.
Есть сборщик мусора, которые вообще не запускается. Например прога одноразовая.
Есть миллисикундый сборщик мусора

### Шаблоны
в лабах будут

C++ не может заменить C из-за отсутствия гарантии имен при создании шаблонов. Мы не знаем, как класс будет называть после компиляции. В C есть гарантии. Механизм, благодаря чему осуществляется гарантия имен называется Манглирование.
В C# класс хранится в шаблоне. При вызове копируется и становится общим для этих типов..

Пример:
```C#
class A<T> {
	T obj; 
}

A<int>a;
A<float>b; //под эти два создаются два разных класса

class C {}
class D {}
A<C>c;
A<D>d; //уже не создаются два разных класса т.к. это один тип
```

### Splay-дерево
После каждой операции с элементом, мы поднимаем этот элемент в корень
1. Вставка: как в BST + splay
2. Поиск: как в BST + splay
3. Удаление: splay от вершины + split & merge

```C#
public class Node <Tkey; Tvalue; TNode>(Tkey key, Tvalue value) where TNode Node<Tkey, Tvalue, TNode> { 
	TNode Left, Right;
}  
```
